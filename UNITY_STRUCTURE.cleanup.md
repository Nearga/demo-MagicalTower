# Unity Structure: Cleanup

## When to read

Read this before deleting files, moving assets, cleaning generated output, changing source-control hygiene, or deciding whether something is project-owned.

## Primary paths

- Source/config: `Assets/`, `Packages/`, `ProjectSettings/`
- Generated/cache: `Library/`, `Logs/`
- Local user/editor state: `UserSettings/`
- Source-control ignore file: parent `.gitignore` at `D:\Local\Projects\Unity\Magical Tower\.gitignore`

## Runtime/source owners

- No project-owned scenes, prefabs, scripts, or runtime assets were found under `Assets/`.
- `ProjectSettings/EditorBuildSettings.asset` has no scene entries, so no build-scene owner is currently registered.

## Data/config owners

- `Packages/manifest.json` and `Packages/packages-lock.json` are project configuration.
- `ProjectSettings/*.asset` files are Unity project configuration.
- `ProjectSettings/VersionControlSettings.asset` uses Visible Meta Files.

## Cross-module routes

- No project-owned code references or serialized scene/prefab references exist yet.
- Future cleanup claims must check both C# references and Unity serialized references before deleting or renaming assets.

## Validation hints

- Generated folders can usually be regenerated by Unity, but do not delete them without explicit cleanup intent.
- For future asset deletion, check `.meta` GUID references in `.unity`, `.prefab`, `.asset`, `.controller`, `.anim`, Addressables, Resources paths, and relevant code references.

## Do-not-touch

- Do not mix cleanup of `Library/`, `Logs/`, or `UserSettings/` with feature work.
- Do not delete `.meta` files independently from their assets.
- Do not treat dependency cache files as project source.

## Open gaps

- Git is rooted one directory above the Unity project. Keep source-control operations anchored at `D:\Local\Projects\Unity\Magical Tower` unless intentionally working inside the nested project path.
